01. Executive Summary

Core Idea

Enable users to get private, per-group summaries delivered into organized topics within their 1-on-1 chat with the bot. Each topic represents one group chat. The user's own credits are consumed. Group members cannot see that someone requested a summary.

Key Capabilities

  • Private Topics — Bot creates topics in the private chat, one per linked group
  • Group Discovery — Bot suggests groups where both the bot and user are members
  • Membership Verification — Real-time check via getChatMember before every summary
  • Credit Consumption — The requesting user's credits are consumed (not the group owner's)
  • Full Feature Parity — All existing parameters work: /summary, /summary 2h, /summary 50
  • Streaming Support — Compatible with Telegram's streaming bot responses in topics

02. Problem Statement

Current Behavior

Today, /summary is invoked inside a group chat and the summary is posted there. This means:

  • All group members see the summary request and output
  • Social pressure — members may feel watched or judged for requesting summaries
  • Summaries clutter the group chat
  • Credits are charged to the group owner (who added the bot), not the person requesting
  • No way to organize summaries from different groups

Desired Behavior

  • User opens their private chat with the bot
  • Each linked group has its own topic (thread) in this private chat
  • User types /summary 2h inside a topic → bot fetches messages from the linked group and delivers the summary into that topic
  • Completely invisible to other group members
  • The requesting user's own credits are consumed

03. Telegram Bot API Research

Key Finding: Private Chat Topics (Bot API 9.4+)

As of Bot API 9.4 (February 2026), bots can create and manage forum topics in private chats. This is enabled via BotFather settings. The bot gets a has_topics_enabled flag, and users see a topic-based interface in their private chat with the bot.

Required API Methods

Method Purpose Key Parameters
createForumTopic Create a topic in the private chat for a linked group chat_id, name, icon_color, icon_custom_emoji_id
editForumTopic Rename topic if group name changes chat_id, message_thread_id, name
deleteForumTopic Remove topic when user unlinks a group chat_id, message_thread_id
closeForumTopic Close topic when user leaves group chat_id, message_thread_id
getChatMember Verify user is still a member of the linked group chat_id, user_id
sendMessage Send summary to the specific topic chat_id, text, message_thread_id

BotFather Configuration Required

One-time Setup in BotFather

The bot must be configured in BotFather with two critical settings:

  1. Open @BotFather → /mybots → select your bot
  2. Go to Bot Settings → Topics in Private Chats → Enable
  3. Go to Bot Settings → Group Privacy → Disable privacy mode (required so the bot can read all group messages, not just commands addressed to it)

Once topics are enabled, all new private chats with the bot will support topics. Existing chats may need the user to restart the conversation.

Note: If the bot is an admin in a group, it receives all messages regardless of privacy mode. But privacy mode must be OFF for groups where the bot is a regular member.

Webhook Update Structure (Private Chat with Topics)

{
  "update_id": 123456789,
  "message": {
    "message_id": 42,
    "message_thread_id": 7,       // ← topic thread ID
    "is_topic_message": true,      // ← confirms it's in a topic
    "from": { "id": 123, "first_name": "User" },
    "chat": { "id": 123, "type": "private" },
    "date": 1234567890,
    "text": "/summary 2h"
  }
}

The message_thread_id field is the key for looking up the linked group in TopicLinksTable.

getChatMember Response Statuses

Status Meaning Allow Summary?
creator User is the group creator Yes
administrator User is a group admin Yes
member Regular member Yes
restricted Restricted but still in group Yes (with is_member: true)
left User has left the group No
kicked User was banned from the group No
Important: Bot Must Be in the Group

The getChatMember API call requires the bot itself to be a member of the target group. The bot cannot verify membership in groups it is not part of. This is consistent with the existing design — the bot is already in the groups and collecting messages.

04. Architecture Decisions

Decision 1: Where Do Topics Live?

Chosen: Private Chat with Topics

Enable forum mode in the bot's private chat. Each user gets topics directly in their 1-on-1 chat.

+ Native Telegram UX, clean topic navigation

+ Built-in Telegram streaming support per topic

+ No extra groups to manage

+ Supported since Bot API 9.4

- Requires BotFather configuration

- Older Telegram clients may not support it

Alternative: Bot-Created Supergroup

Bot creates a private supergroup per user with forum topics enabled.

+ Works on older API versions

- Bots cannot create supergroups via API

- Requires user to manually create the group

- Confusing UX with extra groups

Decision 2: Group Discovery & Linking

Chosen: Bot Suggests Known Groups

Bot queries its own database for groups where both the bot and user are known members, then presents them as inline keyboard buttons.

+ No manual input required from user

+ Bot only shows groups it actually has messages from

+ Clean inline keyboard UX

- Limited to groups where bot has seen the user's messages

Alternative: User Provides Group ID/Link

User manually types a group link or forwards a message.

+ Works for any group

- Poor UX, error-prone

- User needs to know group ID

Decision 3: Credit Model for Private Summaries

Chosen: Requesting User Pays

When a user requests a summary from their private chat, their own credits are consumed, regardless of who owns the group.

+ Fair — each user controls their own usage

+ Group owners aren't drained by others

+ Simple to implement — message.from.id is always the credit owner

Alternative: Group Owner Pays

Credits charged to whoever added the bot to the group (current behavior).

- Unfair for group owners when others use private summaries

- Users could drain someone else's credits

Decision 4: Membership Verification Timing

Chosen: Verify on Every Summary Request

Call getChatMember before generating each summary to ensure the user is still in the group.

+ Always up-to-date

+ No stale membership data

+ Simple implementation

- One extra API call per summary request (~50ms)

Alternative: Cache Membership with TTL

Cache membership status and refresh periodically.

+ Fewer API calls

- Stale data could allow unauthorized access

- Adds caching complexity in Lambda

06. Data Model & DynamoDB Schema

New Table: TopicLinksTable

Maps a user's private chat topic to a group chat. This is the core table for the new feature.

Attribute Type Key Description
userId Number Partition Key Telegram user ID
topicThreadId Number Sort Key The message_thread_id of the topic in the private chat
groupChatId Number — The linked group's Telegram chat ID
groupTitle String — Group name at time of linking (for display)
privateChatId Number — User's private chat ID with the bot
linkedAt Number — Timestamp when the link was created
status String — active | closed (closed when user leaves group)

Global Secondary Index: GroupChatIndex

Attribute Key Type Purpose
groupChatId Partition Key Find all users who linked a specific group
userId Sort Key Uniqueness — one link per user per group

New Table: UserGroupMembership (Cache — Optional)

Tracks which groups a user has been seen in (based on messages stored). Used for the /link discovery flow.

Attribute Type Key Description
userId Number Partition Key Telegram user ID
groupChatId Number Sort Key Group chat the user was seen in
groupTitle String — Last known group title
lastSeenAt Number — Timestamp of last message from user in this group
Design Note: Why a Separate Membership Table?

The existing MessagesTable stores messages by chatId + timestamp. To discover which groups a user belongs to, we'd need a full table scan — expensive and slow. The UserGroupMembership table provides an efficient lookup by userId to find all groups. It's populated passively as messages are stored.

Alternative (simpler, Phase 1): Instead of a new table, query the MessagesTable with a scan + filter for a specific userId. This is inefficient but works for small scale. The dedicated table can be added in Phase 2 if performance becomes an issue.

Existing Tables — No Changes Needed

Table Impact
MessagesTable No changes. Messages are queried by chatId (the group chat ID), which is exactly what we need.
UserCreditsTable No changes. Credits are consumed by userId, works for private summaries.
ChatOwnershipTable No changes. Still tracks who added the bot to groups.

08. Security & Access Control

Critical: Membership Verification

Every private summary request must verify group membership in real-time via getChatMember. Without this, a user could link a group, leave it, and continue accessing summaries of conversations they're no longer part of.

Security Checks (in order)

  1. Topic Link Exists — Verify the topic is linked to a group in DynamoDB
  2. Bot is in Group — The getChatMember call will fail if the bot has been removed from the group. Handle this gracefully.
  3. User is Member — Check status is one of: creator, administrator, member, or restricted with is_member=true
  4. Credits Available — Verify the requesting user has credits remaining

Error Messages

Scenario User-Facing Message
Topic not linked to any group "This topic is not linked to any group. Use /link in the General topic to set one up."
Bot removed from group "I'm no longer a member of [Group Name]. I can't generate summaries for groups I'm not in."
User left/kicked from group "You are no longer a member of [Group Name]. Summary access has been revoked."
Credits exhausted "Daily summary credits have been used up. Credits reset at midnight UTC." (existing message)
Group already linked "[Group Name] is already linked to a topic."

Data Privacy Considerations

  • Summaries are generated from messages already stored by the bot — no new data access
  • Private topics are visible only to the user and the bot
  • No group member notifications when summaries are generated privately
  • UserGroupMembership data is derived from messages already stored (no new tracking)

09. Credits & Usage Integration

Credit Charging Logic

Rule: Private Topic Summary → User Pays

When a /summary command is issued inside a private chat topic, the requesting user's credits are always consumed. This overrides the current group-owner-pays model.

// Updated credit determination logic in SummaryHandler:

let creditOwnerId: number;

if (isPrivateTopicSummary) {
  // Private topic — user always pays their own credits
  creditOwnerId = message.from.id;
} else if (message.chat.type === 'private') {
  // Direct private chat (no topic) — user pays
  creditOwnerId = message.from.id;
} else {
  // Group chat — group owner pays (existing behavior)
  const ownership = await creditsStore.getChatOwner(chatId);
  creditOwnerId = ownership?.ownerUserId ?? message.from.id;
}

Impact on Existing Credit System

Aspect Change
UserCreditsTable No schema changes. Users who only use private summaries will have credit records created automatically.
consumeCredit() No changes to the method. Called with message.from.id instead of group owner ID.
/credits command Should work in private topics — show the user's own credits.
/admin setcredits No changes. Admins can set limits for any user ID.

12. Open Questions & Risks

Q1: Group Discovery — How to Find User's Groups Efficiently?

Challenge: The bot needs to know which groups a user is in. The MessagesTable is keyed by chatId + timestamp, not by userId. Finding all groups for a user requires scanning.

Proposed solutions (in order of preference):

  1. Passive tracking table — Write to UserGroupMembership every time we store a message. Fast lookups, small overhead per message write.
  2. Query from existing data — Use a DynamoDB Global Secondary Index on userId in MessagesTable. Would need schema change and additional RCU.
  3. In-memory scan at link time — Query all groups the bot is in (from ChatOwnershipTable) and call getChatMember for each. Simple but scales poorly.

Recommendation: Option 1 (passive tracking table). Low cost, efficient reads, negligible write overhead.

Q2: Rate Limiting for getChatMember

Challenge: Telegram has rate limits (~30 requests/second for bots). If many users request summaries simultaneously, we could hit the limit.

Mitigation: One getChatMember call per summary request is unlikely to cause issues at current scale. If needed later, implement a short TTL cache (5 minutes) for membership status in Lambda memory or DynamoDB.

Q3: What Happens When the Bot is Removed from a Group?

Scenario: A group admin removes the bot. All users with private topics linked to that group lose access.

Handling: The getChatMember call will fail with "chat not found" or similar error. The bot should catch this, notify the user that the bot is no longer in the group, and offer to unlink the topic.

Q4: Topic Name Conflicts

Scenario: User is in two groups both named "General Discussion".

Handling: Append a short identifier (e.g., last 4 digits of chat ID) to disambiguate: "General Discussion (#4567)".

Q5: Maximum Topics Per Private Chat

Risk: Telegram may impose a limit on the number of topics in a private chat (supergroups support up to ~32,000 topics, but private chats may have a lower limit).

Mitigation: Implement a reasonable client-side limit (e.g., 20 linked groups per user) and test with Telegram to find the actual limit.

Q6: Callback Query Handling

Note: The current handleWebhook only processes update.message. The /link flow requires handling update.callback_query for inline keyboard button presses. This is a new update type that must be added to the handler routing.

Q7: Lambda Cold Start Impact

Adding new DynamoDB clients and Telegram API calls increases cold start time slightly. The singleton caching pattern already in use should be extended to the new stores. Expected impact: ~50-100ms additional cold start time.
